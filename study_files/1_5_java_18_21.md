# Java 18–21 — Virtual Threads, Pattern Matching, Record Patterns

> Java 21 (LTS) با Virtual Threads انقلابی در concurrency برای I/O ایجاد کرد. این بخش پرتکرارترین موضوع مصاحبه‌های مدرن است.

---

## 📖 مفاهیم

### Virtual Threads (Project Loom — Java 21 final)

**توضیح:**

تا قبل از Java 21، هر `Thread` در Java یک thread سیستم‌عامل (platform thread) بود — سنگین (حدود ۱MB استک)، گران برای ساخت، و محدود به چند هزار عدد. مدل کلاسیک «thread-per-request» با این محدودیت برای بار بالا مناسب نبود و توسعه‌دهندگان مجبور به مدل‌های پیچیده‌ی async/reactive شدند.

Virtual Thread یک thread سبک است که توسط JVM مدیریت می‌شود نه OS. میلیون‌ها از آن‌ها قابل ساخت‌اند. وقتی یک virtual thread روی عملیات بلاک‌کننده (I/O) منتظر می‌ماند، JVM آن را از platform thread زیرین (carrier) جدا (unmount) می‌کند و carrier را برای کار دیگری آزاد می‌کند. وقتی I/O تمام شد، دوباره روی یک carrier mount می‌شود. در نتیجه کد بلاک‌کننده‌ی ساده، مقیاس async را می‌دهد.

نکته‌ی حیاتی: virtual threads برای کار **I/O-bound** عالی‌اند اما برای **CPU-bound** هیچ مزیتی ندارند (حتی بدتر، چون سربار اضافه می‌کنند). همچنین `synchronized` می‌تواند باعث «pinning» شود (virtual thread به carrier چسبیده و unmount نمی‌شود) — راه‌حل استفاده از `ReentrantLock` است.

**چرا مهم است:**

مدل ساده‌ی thread-per-request دوباره برای throughput بالا قابل‌استفاده می‌شود. در Spring Boot با `spring.threads.virtual.enabled=true` فعال می‌شود و WebFlux/reactive را برای بسیاری موارد غیرضروری می‌کند.

**مثال کد:**

```java
// ساخت یک virtual thread
Thread.ofVirtual().start(() -> System.out.println("سلام از virtual thread"));

// executor با virtual thread per task — الگوی اصلی
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    List<Future<String>> futures = IntStream.range(0, 10_000)
        .mapToObj(i -> executor.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1)); // I/O شبیه‌سازی‌شده، بلاک‌کننده
            return "task " + i;
        }))
        .toList();
    // ۱۰هزار تسک تقریباً همزمان با چند carrier thread اجرا می‌شوند
}
```

**نکات کلیدی:**

- virtual threads برای I/O-bound است نه CPU-bound.
- `synchronized` می‌تواند pinning ایجاد کند؛ برای بخش‌های بلاک‌کننده از `ReentrantLock` استفاده کنید.
- pool کردن virtual threads بی‌معناست؛ آن‌ها ارزان و یک‌بارمصرف‌اند (per-task).

### Pattern Matching for switch (Java 21 final)

**توضیح:**

switch می‌تواند بر اساس **نوع** match کند (نه فقط مقدار). با sealed types، کامپایلر exhaustiveness را تضمین می‌کند. می‌توان **guard** (`when`) برای شرط اضافه گذاشت و `null` را به‌صورت صریح handle کرد.

**مثال کد:**

```java
sealed interface Shape permits Circle, Rectangle {}
record Circle(double r) implements Shape {}
record Rectangle(double w, double h) implements Shape {}

static String describe(Object obj) {
    return switch (obj) {
        case null -> "خالی";
        case Circle c when c.r() > 10 -> "دایره‌ی بزرگ"; // guard
        case Circle c -> "دایره";
        case Rectangle r -> "مستطیل";
        default -> "ناشناخته";
    };
}
```

**نکات کلیدی:**

- `when` برای guard شرطی استفاده می‌شود.
- ترتیب caseها مهم است: حالت خاص‌تر باید قبل از حالت عام‌تر بیاید.
- `null` را می‌توان به‌صراحت در switch handle کرد.

### Record Patterns (Java 21 final)

**توضیح:**

destructuring (تجزیه) recordها در `instanceof` و `switch`. به‌جای دسترسی به componentها با accessor، می‌توان مستقیماً آن‌ها را به متغیر bind کرد، حتی به‌صورت تو در تو (nested).

**مثال کد:**

```java
record Point(int x, int y) {}
record Line(Point start, Point end) {}

static String describe(Object obj) {
    return switch (obj) {
        // destructuring تو در تو
        case Line(Point(var x1, var y1), Point(var x2, var y2)) ->
            "خط از (%d,%d) تا (%d,%d)".formatted(x1, y1, x2, y2);
        case Point(var x, var y) -> "نقطه (%d,%d)".formatted(x, y);
        default -> "نامشخص";
    };
}
```

**نکات کلیدی:**

- destructuring تو در تو کد را بسیار خواناتر می‌کند.
- با sealed + switch ترکیب قدرتمندی برای پردازش داده می‌سازد.

### Sequenced Collections (Java 21)

**توضیح:**

تا قبل از این، دسترسی به اولین/آخرین عنصر در مجموعه‌ها ناهماهنگ بود (`list.get(0)` در برابر `deque.getFirst()`). interfaceهای جدید `SequencedCollection`, `SequencedSet`, `SequencedMap` API یکنواخت می‌دهند: `getFirst()`, `getLast()`, `addFirst()`, `addLast()`, `reversed()`.

**مثال کد:**

```java
SequencedCollection<String> list = new ArrayList<>(List.of("a", "b", "c"));
System.out.println(list.getFirst()); // a
System.out.println(list.getLast());  // c
System.out.println(list.reversed()); // [c, b, a]

SequencedMap<String, Integer> map = new LinkedHashMap<>();
map.putFirst("first", 1);
map.putLast("last", 2);
```

**نکات کلیدی:**

- API یکنواخت برای اولین/آخرین عنصر و معکوس‌سازی.
- `reversed()` یک view است نه کپی.

### Structured Concurrency (Java 21 — preview)

**توضیح:**

مدلی که چند تسک هم‌زمان مرتبط را به‌عنوان یک واحد مدیریت می‌کند: اگر یکی شکست بخورد، بقیه لغو می‌شوند؛ scope تا اتمام همه منتظر می‌ماند. این از thread leak و خطاهای پراکنده جلوگیری می‌کند و error handling را شفاف می‌کند.

**مثال کد:**

```java
// Java 21 preview API
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    Subtask<User> user = scope.fork(() -> fetchUser(id));
    Subtask<Order> order = scope.fork(() -> fetchOrder(id));
    scope.join().throwIfFailed(); // منتظر همه؛ اگر یکی شکست خورد لغو بقیه
    return new Dashboard(user.get(), order.get());
}
```

**نکات کلیدی:**

- خطای یک تسک باعث لغو بقیه می‌شود (fail-fast).
- چرخه‌ی حیات تسک‌ها به scope گره می‌خورد → بدون leak.

## 🎯 سوالات مصاحبه

### سوال ۱: Virtual Threads چه مشکلی را حل می‌کنند و کِی مناسب‌اند؟

**سطح:** Senior / Lead
**تکرار:** خیلی زیاد

**جواب کامل:**

مشکل اصلی: platform threadها گران و محدودند، پس مدل thread-per-request برای بار بالای I/O مقیاس نمی‌گرفت و توسعه‌دهندگان مجبور به async/reactive پیچیده می‌شدند. Virtual threads thread سبک مدیریت‌شده توسط JVM هستند؛ هنگام بلاک شدن روی I/O، carrier آزاد می‌شود و میلیون‌ها از آن‌ها ممکن است. در نتیجه کد ساده و خطی (blocking style) با مقیاس async نوشته می‌شود.

مناسب برای: workload‌های I/O-bound با concurrency بالا (فراخوانی DB، HTTP، صف). نامناسب برای: CPU-bound (مزیتی ندارد چون باز هم به تعداد هسته محدودید). نکته‌ی مهم: `synchronized` روی بخش بلاک‌کننده می‌تواند pinning ایجاد کند که مزیت را خنثی می‌کند؛ راه‌حل `ReentrantLock`.

**کد توضیحی:**

```java
// Spring Boot: application.yml
// spring.threads.virtual.enabled: true
// هر request روی یک virtual thread اجرا می‌شود
```

**نکته مصاحبه:**

تمایز Senior/Lead: دانستن pinning، تفاوت I/O-bound و CPU-bound، و اینکه virtual threads بسیاری از موارد استفاده‌ی WebFlux را غیرضروری می‌کند. Follow-up: «چرا نباید virtual threads را pool کرد؟»

---

### سوال ۲: تفاوت Virtual Thread با Platform Thread چیست؟

**سطح:** Senior
**تکرار:** زیاد

**جواب کامل:**

Platform thread یک wrapper نازک روی thread سیستم‌عامل است: ساخت گران، استک بزرگ (~1MB)، تعداد محدود (هزاران). Virtual thread توسط JVM مدیریت می‌شود: ساخت ارزان، استک قابل‌رشد در heap، تعداد میلیونی. چند virtual thread روی تعداد کمی carrier (platform thread) multiplexed می‌شوند. هنگام بلاک شدن I/O، virtual thread از carrier جدا می‌شود (mount/unmount). از دید کد، API یکسان است (`Thread`).

**نکته مصاحبه:**

Follow-up: «scheduler virtual threads چیست؟» (پاسخ: یک ForkJoinPool اختصاصی به‌عنوان carrier pool).

---

### سوال ۳: pinning چیست و چطور از آن جلوگیری می‌کنیم؟

**سطح:** Lead
**تکرار:** متوسط

**جواب کامل:**

pinning زمانی رخ می‌دهد که یک virtual thread نمی‌تواند از carrier خود unmount شود، پس carrier در طول عملیات بلاک‌کننده اشغال می‌ماند و مزیت مقیاس‌پذیری از بین می‌رود. دلایل اصلی: اجرای کد بلاک‌کننده داخل بلوک `synchronized`، یا فراخوانی native method (JNI). راه‌حل: جایگزینی `synchronized` با `ReentrantLock` در مسیرهای بلاک‌کننده. می‌توان pinning را با `-Djdk.tracePinnedThreads=full` رصد کرد.

**نکته مصاحبه:**

Lead به ابزار تشخیص و راه‌حل عملی اشاره می‌کند. Follow-up: «آیا در Java نسخه‌های بعد pinning با synchronized حل شده؟» (در JDK‌های جدیدتر بهبود یافته اما اصل احتیاط می‌ماند).

---

### سوال ۴: Virtual Threads در برابر Reactive (WebFlux) — کدام؟

**سطح:** Lead
**تکرار:** زیاد

**جواب کامل:**

هر دو مشکل مقیاس‌پذیری I/O را حل می‌کنند اما با مدل ذهنی متفاوت. Reactive (Reactor) با callback/operator و backpressure کار می‌کند؛ قدرتمند اما یادگیری سخت، دیباگ دشوار، و آلودگی کل stack. Virtual threads کد بلاک‌کننده‌ی ساده و خطی را حفظ می‌کنند و debugging/stacktrace طبیعی است. برای اکثر برنامه‌های CRUD/microservice، virtual threads ساده‌تر و کافی است. Reactive هنوز برای streaming، backpressure واقعی، و ترکیب پیچیده‌ی جریان‌ها مزیت دارد.

**نکته مصاحبه:**

Lead trade-off را می‌فهمد و mindlessly یکی را تجویز نمی‌کند. Follow-up: «backpressure را virtual threads چطور مدیریت می‌کند؟» (با محدود کردن concurrency مثل Semaphore، نه به‌صورت داخلی).

---

### سوال ۵: Record Patterns چه چیزی را ساده می‌کنند؟

**سطح:** Senior
**تکرار:** متوسط

**جواب کامل:**

destructuring recordها در switch/instanceof، به‌خصوص ساختارهای تو در تو. به‌جای match کردن نوع و سپس فراخوانی accessorها، componentها مستقیماً به متغیر bind می‌شوند. با sealed types و switch، این الگوی data-oriented programming را می‌سازد که در آن منطق بر اساس شکل داده نوشته می‌شود و کامپایلر پوشش کامل را تضمین می‌کند.

**نکته مصاحبه:**

Follow-up: «تفاوت با visitor pattern چیست؟» (pattern matching جایگزین مختصر و type-safe برای visitor در بسیاری موارد است).

## ⚠️ اشتباهات رایج

### اشتباه ۱: pool کردن virtual threads

```java
// ❌ بی‌معنا و مضر
ExecutorService pool = Executors.newFixedThreadPool(200);
// سپس virtual thread داخلش...
```

```java
// ✅ per-task executor
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(task);
}
```

**توضیح:** virtual threads ارزان و یک‌بارمصرف‌اند؛ pool کردن مزیت‌شان را از بین می‌برد.

---

### اشتباه ۲: استفاده از virtual threads برای کار CPU-bound

```java
// ❌ هیچ مزیتی ندارد، فقط سربار
executor.submit(() -> heavyMatrixMultiplication());
```

```java
// ✅ برای CPU-bound از pool به اندازه‌ی هسته‌ها
var cpuPool = Executors.newFixedThreadPool(Runtime.getRuntime().availableProcessors());
```

**توضیح:** virtual threads فقط برای I/O-bound مزیت دارند.

---

### اشتباه ۳: synchronized در مسیر بلاک‌کننده‌ی virtual thread

```java
// ❌ pinning
synchronized (lock) {
    jdbcCall(); // بلاک‌کننده، carrier اشغال می‌ماند
}
```

```java
// ✅
lock.lock();
try { jdbcCall(); } finally { lock.unlock(); }
```

**توضیح:** `synchronized` روی I/O باعث pinning می‌شود؛ `ReentrantLock` unmount را اجازه می‌دهد.

---

### اشتباه ۴: گذاشتن case عام قبل از خاص در pattern switch

```java
// ❌ case Circle هرگز نمی‌رسد چون عام‌تر زودتر match می‌شود
switch (shape) {
    case Shape s -> "shape";
    case Circle c -> "circle"; // unreachable
}
```

```java
// ✅ خاص قبل از عام
switch (shape) {
    case Circle c -> "circle";
    case Shape s -> "shape";
}
```

**توضیح:** ترتیب caseها از خاص به عام باشد.

## 🔗 ارتباط با سایر مفاهیم

- Virtual Threads با **Spring Boot** (`spring.threads.virtual.enabled`)، **Concurrency** و **Reactive** فصل‌های دیگر گره خورده‌اند.
- Pattern Matching + Record Patterns + Sealed = data-oriented programming که با **Design Patterns** (جایگزین visitor) مرتبط است.
- Structured Concurrency با **resilience patterns** (timeout، cancellation) ترکیب می‌شود.
- این مفاهیم پایه‌ی تصمیم معماری **MVC vs WebFlux** در بخش Spring هستند.
