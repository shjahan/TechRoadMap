# React 19 — Hooks، Server Components، State Management، Performance

> برای نقش‌های full-stack، درک React مدرن لازم است. Hooks و React 19 features موضوعات کلیدی‌اند.

---

## 📖 مفاهیم

### مبانی — JSX، Virtual DOM، Components

**توضیح:**

React یک کتابخانه‌ی UI declarative است. **JSX** سینتکس شبیه HTML در JS. **Virtual DOM** یک نمایش in-memory از UI؛ React تغییرات را با **reconciliation** (diff بین VDOM قدیم و جدید) محاسبه و فقط بخش‌های تغییریافته را به DOM واقعی اعمال می‌کند. **Functional Components** (استاندارد امروز) با Hooks، در برابر Class Components (legacy). **Props** immutable و read-only؛ **State** با `useState`.

**نکات کلیدی:**

- functional components + hooks استاندارد مدرن است.
- props را تغییر ندهید (immutable)؛ state را با setter به‌روز کنید.

---

### Hooks

**توضیح:**

- **پایه:** `useState` (state محلی)، `useEffect` (side-effect، lifecycle)، `useContext` (دسترسی به context)، `useRef` (مرجع پایدار/DOM)، `useReducer` (state پیچیده).
- **Performance:** `useMemo` (memoize محاسبه)، `useCallback` (memoize تابع) — با React Compiler کمتر نیاز.
- **جدید (18/19):** `useTransition` (به‌روزرسانی غیرفوری)، `useDeferredValue`، `useId`، `use()` (خواندن async)، `useOptimistic` (optimistic UI)، `useFormStatus`.

قوانین Hooks: فقط در سطح بالای component (نه در حلقه/شرط)، فقط در functional component.

**مثال کد:**

```jsx
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  useEffect(() => {
    let cancelled = false;
    fetchUser(userId).then(u => { if (!cancelled) setUser(u); });
    return () => { cancelled = true; }; // cleanup برای جلوگیری از race
  }, [userId]); // dependency array
  return user ? <div>{user.name}</div> : <Loading />;
}
```

**نکات کلیدی:**

- `useEffect` dependency array را درست بدهید (وگرنه stale closure یا loop).
- cleanup function برای جلوگیری از memory leak و race.
- Hooks فقط در سطح بالا (rules of hooks).

---

### React 19 Features

**توضیح:**

- **React Compiler:** auto-memoization؛ دیگر نیازی به `useMemo`/`useCallback` دستی نیست.
- **Server Components (RSC):** render روی server، بدون ارسال JS به client → bundle کوچک‌تر.
- **Server Actions:** تابع روی server که از client صدا زده می‌شود (mutation).
- **Actions API:** ترکیب mutation + pending state + error.
- **Document Metadata:** `<title>`/`<meta>` مستقیم در component.

**نکات کلیدی:**

- React Compiler memoization دستی را کم می‌کند.
- Server Components حجم JS سمت client را کاهش می‌دهند.

---

### State Management

**توضیح:**

- Local: `useState`, `useReducer`.
- Shared: Context API (برای جلوگیری از props drilling، اما برای state پرتغییر مناسب نیست — re-render زیاد).
- Global: **Zustand** (ساده)، **Redux Toolkit** (پیچیده، با devtools)، Jotai، Recoil.
- **Server State: TanStack Query (React Query)** — caching، revalidation، background sync برای داده‌ی سرور (جدا از client state).

نکته‌ی مهم: server state (داده از API) را با client state (UI state) قاطی نکنید؛ TanStack Query برای server state است.

**نکات کلیدی:**

- Context برای state کم‌تغییر؛ برای پرتغییر re-render مشکل می‌سازد.
- server state را با TanStack Query مدیریت کنید نه Redux.

---

### Performance

**توضیح:**

- **Code Splitting:** `React.lazy()` + `Suspense` برای بارگذاری lazy.
- **`React.memo`:** جلوگیری از re-render بی‌مورد.
- **Virtualization:** `react-window`/TanStack Virtual برای لیست بزرگ (فقط آیتم‌های visible render شوند).
- **Profiler:** React DevTools برای یافتن re-render اضافه.

**نکات کلیدی:**

- virtualization برای لیست هزاران آیتمی.
- با React Compiler، memoization دستی کمتر لازم.

## 🎯 سوالات مصاحبه

### سوال ۱: Virtual DOM و reconciliation چطور کار می‌کنند؟

**سطح:** Senior
**تکرار:** زیاد

**جواب کامل:**

Virtual DOM یک نمایش سبک in-memory از UI است. وقتی state تغییر می‌کند، React یک VDOM جدید می‌سازد و آن را با نسخه‌ی قبلی **diff** می‌کند (reconciliation). به‌جای به‌روزرسانی کل DOM واقعی (که گران است)، فقط تفاوت‌ها را به DOM واقعی اعمال می‌کند. برای کارایی diff، React از heuristic استفاده می‌کند: عناصر با type متفاوت کاملاً جایگزین می‌شوند، و برای لیست‌ها از **key** برای تشخیص جابه‌جایی/اضافه/حذف استفاده می‌کند. به همین دلیل key درست (یکتا و پایدار، نه index آرایه) برای performance و درستی حیاتی است.

**نکته مصاحبه:**

Senior به اهمیت key (نه index) در reconciliation اشاره می‌کند. Follow-up: «چرا index به‌عنوان key بد است؟» (با reorder/حذف باگ می‌سازد).

---

### سوال ۲: مشکلات رایج `useEffect`؟

**سطح:** Senior
**تکرار:** زیاد

**جواب کامل:**

مشکلات: (۱) **dependency array اشتباه** — جا انداختن یک dependency باعث stale closure (مقدار قدیمی) می‌شود؛ array خالی `[]` یعنی فقط mount. (۲) **infinite loop** — اگر effect یک state را set کند که خودش dependency است. (۳) **race condition** — اگر دو fetch همزمان شروع شوند، نتیجه‌ی قدیمی ممکن آخر برسد؛ باید با cleanup یا AbortController لغو شود. (۴) **memory leak** — subscription/timer بدون cleanup. (۵) استفاده‌ی بیش از حد — بسیاری از کارها (مثل derived state) نیازی به effect ندارند. روند مدرن: برای fetch از TanStack Query به‌جای useEffect دستی استفاده کنید.

**نکته مصاحبه:**

Senior به race condition و cleanup اشاره می‌کند و می‌داند fetch بهتر با React Query است.

---

### سوال ۳: client state در برابر server state؟

**سطح:** Senior
**تکرار:** متوسط

**جواب کامل:**

client state داده‌ای است که فقط در مرورگر زندگی می‌کند (UI state: مودال باز است، تب فعال، فرم). server state داده‌ای است که source of truth آن سرور است (لیست کاربران، محصولات) و باید fetch، cache، sync، و invalidate شود. اشتباه رایج: مدیریت server state با Redux که نیاز به نوشتن دستی loading/error/caching/refetch دارد. ابزار مخصوص server state مثل **TanStack Query** این‌ها را خودکار می‌کند: caching، background refetch، stale-while-revalidate، deduplication، و optimistic update. قاعده: client state با useState/Zustand، server state با TanStack Query.

**نکته مصاحبه:**

Senior جداسازی این دو و ابزار درست هر کدام را می‌داند.

---

### سوال ۴: React Server Components چه مزیتی دارند؟

**سطح:** Senior / Lead
**تکرار:** متوسط

**جواب کامل:**

RSC روی server render می‌شوند و **هیچ JavaScript به client نمی‌فرستند** (برخلاف SSR سنتی که HTML می‌فرستد اما سپس hydrate می‌کند با JS). مزایا: bundle JS کوچک‌تر (کد component و وابستگی‌هایش به client نمی‌رود)، دسترسی مستقیم به منابع server (DB، fs) بدون API، و بهبود initial load. Server Components نمی‌توانند state یا event handler داشته باشند (interactivity)؛ برای آن از Client Components (`'use client'`) استفاده می‌کنید. ترکیب RSC (محتوای static/data-heavy) + Client Components (تعاملی) معماری بهینه می‌دهد. این پایه‌ی App Router در Next.js است.

**نکته مصاحبه:**

Senior تفاوت RSC با SSR سنتی (no JS در برابر hydration) را می‌داند.

## ⚠️ اشتباهات رایج

### اشتباه ۱: index به‌عنوان key

```jsx
// ❌ با reorder/حذف باگ
{items.map((item, i) => <Item key={i} />)}
```

```jsx
// ✅ شناسه‌ی پایدار
{items.map(item => <Item key={item.id} />)}
```

**توضیح:** index با تغییر ترتیب لیست reconciliation را خراب می‌کند.

---

### اشتباه ۲: dependency array اشتباه در useEffect

```jsx
// ❌ stale closure یا loop
useEffect(() => { doSomething(value); }, []); // value جا افتاده
```

```jsx
// ✅
useEffect(() => { doSomething(value); }, [value]);
```

**توضیح:** dependency جاافتاده = مقدار قدیمی؛ dependency اضافه = اجرای بی‌مورد.

---

### اشتباه ۳: server state در Redux

```jsx
// ❌ مدیریت دستی fetch/cache در Redux
```

```jsx
// ✅ TanStack Query
const { data } = useQuery({ queryKey: ['users'], queryFn: fetchUsers });
```

**توضیح:** server state ابزار مخصوص می‌خواهد.
